Open a pull request. The agent writes the title AND body; the CLI
handles the mechanics (branch push + `gh pr create`).

## Arguments

Use `$ARGUMENTS` to determine the target branch.

- Target branch: `$1` (default: `main`)

## Emoji map — change-type → emoji

Pick the emoji from the branch prefix (or, if ambiguous, from the
dominant commit type). One emoji per title, always at the start:

| Type | Emoji | Type | Emoji |
|---|---|---|---|
| feat | ✨ | docs | 📝 |
| fix | 🐛 | test | 🧪 |
| refactor | ♻️ | chore | 🔧 |
| perf | ⚡ | style | 🎨 |
| ci | 🚀 | revert | ⏪ |
| improve / enhance | 💫 | | |

## Instructions

1. **Resolve target branch.** If `$1` is empty, run
   `git branch -r | grep -v HEAD` and ask the user which branch to
   target (default: `main`).

2. **Resolve the PR-body language** by consulting, in order:
   - `.octopus.yml` → `language.code` (per-scope form).
   - `.octopus.yml` → `language:` (short form, applies to all scopes).
   - Cascade from `rules/common/language.md` (docs/, `git log`, locales/).
   - Default: English.
   Apply the resolved value to every line of the body AND the title.

3. **Collect context.** Let `BASE` be the target branch.
   - Commits: `git log ${BASE}..HEAD --format='%H%n%s%n%b%n---%n'`.
   - File stats: `git diff --stat ${BASE}..HEAD`.
   - Semantic diff: `git diff ${BASE}..HEAD` — scan for the *meaning*
     of the change, not line-by-line.

4. **Detect references** (populate the `🔗 References` section only
   if at least one hit). Scan the branch name, commit messages, and
   diff for:
   - Roadmap IDs matching `RM-\d+` → link to `docs/roadmap.md`
     (append `#<id>` anchor when the roadmap has a matching entry).
   - Jira-style tracker IDs matching `[A-Z]{2,}-\d+`. Exclude
     obvious false positives: HTTP codes (HTTP-200, 4xx/5xx), ISO
     standards (ISO-8601), encoding (UTF-8), RFC-NNNN, and single
     letters (`A-1`). When uncertain, check whether the surrounding
     text reads like a ticket mention.
   - Notion / ad-hoc trackers: full URLs
     (`https://www.notion.so/…`, `https://github.com/…/issues/…`)
     or `[tracker: <value>]` markers in commit messages.
   - Local specs/ADRs cited in commits: `docs/specs/*.md`,
     `docs/adr/*.md`.
   Omit the entire section if nothing was detected.

5. **Craft the title.**
   - Pick the type from the branch prefix (`feat/…` → ✨,
     `fix/…` → 🐛, …). If the branch prefix is ambiguous, fall
     back to the dominant commit type.
   - Write a short human-friendly title that communicates the
     outcome — NOT a commit log. Example:
     `🐛 Fix: octopus update ignores latest remote when metadata exists`.
   - Do not use the Conventional-Commits `type:` prefix. The emoji
     already conveys the intent.
   - Stay under 72 characters when possible.

6. **Read the template.** Open `cli/pr-body-default.md`. Follow its
   section contract and the hard rules in its header comment.

7. **Write the body to a tempfile.**
   - Create a file with `mktemp --suffix=.md` (store the path).
   - Sections, in order: `## 📦 What`, `## 💡 Why` (omit if
     self-evident), `## ✅ Test plan`, `## 🔗 References`
     (omit if empty), collapsed `## 📂 Files changed`.
   - End with `<sub>🐙 generated by Octopus</sub>` on its own line,
     preceded by a `---` horizontal rule.
   - Write every line in the resolved language. Do not mix
     languages. Do not default to the conversation language.
   - Never dump `git log` as a bullet list.
   - Never list files outside the `<details>` block.

8. **Open the PR** by running:
   ```
   octopus pr-open --target <branch> --body-file <tempfile> --title "<emoji> <title>"
   ```
   The CLI aborts if `--body-file` is missing — always pass it.
   Pass `--title` even when it mirrors the fallback; that makes the
   shipped title explicit.

9. **Capture the PR number** from the output line
   `OCTOPUS_PR=<number>`.

10. **Show the submitted title + body.** Run:
    ```
    gh pr view <number> --json url,title,body -q '"URL: \(.url)\n\nTitle: \(.title)\n\n## PR Body\n\(.body)"'
    ```
    Display the full URL, title, and body so the user can verify
    what was submitted.

11. **Close out.** Suggest `/octopus:pr-review <number>` as the next
    step.
