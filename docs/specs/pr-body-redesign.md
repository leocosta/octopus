# Spec: PR body redesign — agent-driven narrative + language scope fix

**Status:** Implemented (2026-04-20)
**Date:** 2026-04-20
**Related:** `commands/pr-open.md`, `cli/lib/pr-open.sh`,
`cli/pr-body-default.md`, `.octopus.example.yml`,
`cli/lib/setup-wizard.sh`

## Problem

The current PR body generator (`cli/lib/pr-open.sh → generate_pr_body`)
produces a wall of text that reads like a dumb changelog:

- Every commit is dumped title + body as a bullet list.
- Every changed file is listed inline with icons.
- A "Key Changes" heuristic greps for `function|class|const|let|var`
  and surfaces meaningless names (PR #60 ended up listing `prefixes`,
  `list`, `restates` — variable names pattern-matched inside unrelated
  prose).
- "How to Test" is generic boilerplate ("review the diff for
  correctness").
- No trace of agency: reviewers can't tell *why* the change exists or
  *what* to exercise.

Secondary problem: `.octopus.yml` scopes commit messages and PR
descriptions under `language.docs`, but they are developer-facing
artefacts describing code — same audience as code comments. They
belong under `language.code`.

## Goals

1. PR bodies become objective, narrative summaries — three sections,
   written by the agent, not a log dump.
2. Every Octopus-generated PR carries a subtle signature (🐙).
3. `language.code` becomes the canonical scope for commit messages
   and PR descriptions.
4. `cli/lib/pr-open.sh` stops generating text — it handles mechanics
   only (push, `gh pr create`, capture PR number).

## Non-goals

- Migrating existing PRs or rewriting historical bodies.
- Changing how commit messages are produced (already agent-driven in
  the dev-flow).
- Touching `pr-comments`, `pr-review`, or `pr-merge` — they don't
  generate PR text.
- Auto-migrating `.octopus.yml` files in consumer repos — the field
  names are unchanged, only their semantics shift.

## Design

### Architecture

Responsibility split:

- **`cli/lib/pr-open.sh`** — mechanical only: parse flags, push the
  branch, read `.octopus.yml`, invoke `gh pr create --body-file`,
  extract the PR number. **No text generation.** If invoked without
  `--body-file`, aborts with
  `ERROR: PR body is required; invoke via /octopus:pr-open command`.
- **`commands/pr-open.md`** — instructs the agent: read language
  config, collect commits + diff, synthesise the body following
  `cli/pr-body-default.md`, save to a tempfile, call
  `octopus pr-open --body-file <path>`.
- **`cli/pr-body-default.md`** — rewritten from a fill-in placeholder
  to a semantic guide (rules + section skeleton) the agent follows.
- **Agent** — writes the narrative, applies the resolved language,
  appends the signature.

### Manifest schema correction

`.octopus.yml` `language:` scopes now mean:

| Scope | Covers |
|---|---|
| `code` | identifiers (always `en`), code comments, **commit messages, PR descriptions** |
| `docs` | specs, ADRs, RFCs, README |
| `ui` | user-facing messages, app copy |

The field names are unchanged; only the semantic boundary moves.
Short form `language: <code>` continues to apply to all scopes.

Files updated:

- `.octopus.example.yml` — comments on lines 21–30.
- `cli/lib/setup-wizard.sh:771` — prompt text for `docs` scope:
  drop `commits, PRs`.
- `cli/lib/setup-wizard.sh:774` — prompt text for `code` scope:
  add `commits, PRs`.

No parsing code changes — the fields already exist.

### Language resolution (PR body)

Order applied by the command before invoking the agent:

1. `.octopus.yml` → `language.code` (per-scope form).
2. `.octopus.yml` → `language:` (short form).
3. Cascade from `rules/common/language.md` (docs/, git log, locales).
4. Default: English.

Resolved value is passed explicitly in the agent prompt: *"write the
PR body in `<lang>`"*, so the agent does not drift to the
conversation language.

### PR body template (`cli/pr-body-default.md`)

Rewritten as a semantic guide, not a placeholder. Contains:

- A top-level HTML comment explaining audience, language rules, and
  section contract.
- Three required sections — **What**, **Why**, **Test plan** — each
  with inline guidance comments.
- A `<details>` block for the file list, with guidance on grouping
  by change type and collapsing siblings via brace syntax (e.g.
  `skills/{a,b,c}/SKILL.md`).
- Footer: `<sub>🐙 generated by Octopus</sub>`, always last.

Hard rules the agent follows:

- Never dump `git log` as bullet list.
- Never list files outside the `<details>` block.
- Signature is fixed and always last.
- Omit a section if there is nothing substantive to say (except
  **What**, which is always present).
- If no test plan applies, state the reason (e.g. "docs-only change,
  no tests").

### `/octopus:pr-open` flow

```
1. Read .octopus.yml → language.code (or short form); else cascade.
2. Collect:
   - git log <base>..HEAD --format="%H %s%n%b"
   - git diff --stat <base>..HEAD
   - git diff <base>..HEAD (for semantic reading)
   - Roadmap / spec references (RM-NNN, docs/specs/*) cited in commits.
3. Synthesise the body following cli/pr-body-default.md.
4. Save to $(mktemp --suffix=.md).
5. Run: octopus pr-open --target <branch> --body-file <path>.
6. Capture OCTOPUS_PR=<number> from output.
7. Show URL + submitted body (gh pr view) to the user.
```

### Shell simplification

Lines 43–183 of `cli/lib/pr-open.sh` — `generate_pr_body`,
`emoji_for_commit`, and the heuristic blocks — are removed. What
remains (~40 lines): flag parsing, branch push, `gh pr create`,
PR-number extraction.

## Testing

Extend `tests/test_pr_open.sh` (create if missing):

1. `octopus pr-open` without `--body-file` aborts with the required
   error message.
2. `octopus pr-open --body-file <missing>` aborts with
   `Body file not found`.
3. Given a fixture body file containing the signature line, assert
   `gh pr create` is invoked with that file (using a `gh` stub).
4. In `tests/test_parse_yaml.sh` (or equivalent): assert `language.code`
   and `language.docs` round-trip through the manifest parser.

No test exercises the agent — narrative quality is validated by the
human self-review in dev-flow Step 3.

## Migration notes

- Consumer repos with `language.docs: pt-br` and no `language.code`
  will see their PR bodies move from Portuguese to the
  `rules/common/language.md` cascade result (likely the same, but
  not guaranteed). Documented as a minor-bump breaking semantic
  change in the CHANGELOG.
- No auto-migration tooling. The field names are stable; only their
  meaning shifts.
- Repos using the short form (`language: pt-br`) are unaffected —
  it still applies to every scope.

## Risks

- **Language drift.** The agent may occasionally write in the
  conversation language instead of the resolved scope. Mitigated by
  passing the resolved value explicitly in the command prompt and
  covering it in the agent instruction block of `commands/pr-open.md`.
- **Template bloat.** Authors may feel tempted to add a fourth or
  fifth section. The spec explicitly enumerates the three and the
  agent's hard rules reject extras. Enforced socially, not by code.
- **Lost determinism.** The current shell generator is reproducible;
  the agent is not. Accepted trade-off — the objective is quality,
  not reproducibility, and the self-review step shows the result
  before it ships.

## Out of scope

- Machine-checking that PRs contain the signature (possible
  follow-up as a GitHub Action).
- Multi-language PRs for the same repo (not a real use case today).
- Auto-translating existing PR bodies.
