# PRD — Local Guardrails: Quality, Style & Semantic Grounding

- **Roadmap item:** RM-088
- **Status:** **shipped** — the in-scope solution landed in v1.69.0 (#106) as the `audit-grounding` skill + `grounding-check` Stop hook (semantic signal) over the existing `guardrails` bundle (syntactic block). The two **Out of Scope** failure modes below (non-existent APIs / missing files, and "claimed done without running") were deferred and now seed **RM-111** (Cluster 20).
- **Bundle:** `quality` (the `audit-grounding` skill registers here, beside the other `audit-*` skills — no loose skill); the deterministic syntactic layer is shipped by the `guardrails` bundle
- **Source:** `/octopus:interview` → `/octopus:doc-prd` session, 2026-05-28

---

## Problem

Teams that consume Octopus in their product repositories have no
enforcement layer that survives a forgetful developer or an AI agent
that skips a step. Three failure modes leak through today: code that
is unformatted or untyped reaches version control; team styling
conventions drift because nothing checks them at commit time; and AI
agents produce two kinds of semantic hallucination — inventing a team
convention nobody agreed on, and asserting a domain fact that is
false. The first two are syntactic and objectively checkable; the
last two are semantic and can only be judged against the team's
living source of truth, which already exists in the consumer repo
(generated by Octopus inputs: the domain context, decisions of
record, and the knowledge base) but is never automatically consulted.

## Solution

Ship a local-only guardrail layer that consumer repos inherit by
enabling one bundle. It enforces the objective, syntactic concerns
with a deterministic block at commit time — catching every commit,
whether written by a human or an agent — and surfaces the semantic
concerns as a non-blocking signal at the end of every agent task,
comparing what the agent produced against the living source of truth
and reporting divergences for a human to judge. Nothing runs in CI or
server-side; the entire layer is local.

## User Stories

**Developer committing code (with or without an AI agent)**
- As a developer, when I try to commit unformatted code, the commit
  is blocked before it leaves my machine, with a message naming what
  to fix.
- As a developer, when I try to commit code that fails the type
  check, the commit is blocked.
- As a developer, when I try to commit a hardcoded secret, the commit
  is blocked.
- As a developer, when I try to commit code that violates a styling
  convention the formatter can enforce, it is auto-corrected or the
  commit is blocked.
- As a developer who wrote code entirely by hand with no AI involved,
  the same syntactic gate still catches me — the guarantee does not
  depend on an agent session having happened.

**AI agent finishing a task**
- As an agent, when I finish a task, a review step runs automatically
  without me having to remember to invoke it.
- As an agent, when I have introduced a naming or structural
  convention that is not in the source of truth, the review surfaces
  it as a divergence rather than letting it pass silently.
- As an agent, when I have asserted a domain fact that contradicts or
  is absent from the decisions of record, the review flags it.

**Developer reviewing the agent's output**
- As a developer, at the end of an agent task I see a clear signal
  listing each claim or convention that diverges from the source of
  truth, and I decide whether to accept or correct it — the guardrail
  never blocks me on a semantic judgment.
- As a developer, when the agent's work is fully grounded, the review
  produces no noise.

**Maintainer of the source of truth**
- As a maintainer, the guardrail reads the same domain context,
  decisions of record, and knowledge base I already curate, so
  keeping those fresh directly improves the signal quality.

**Consumer-repo adopter**
- As an adopter, I enable the guardrails by adding one bundle to my
  Octopus config, and the hooks and the review skill are wired up at
  setup time without further manual steps.

## Implementation Decisions

- **Two trigger layers, both local.** A pre-commit trigger owns the
  syntactic gate so it catches every commit including hand-written,
  non-AI code. A session-stop trigger owns the semantic review so it
  fires at the end of every agent task. Neither is optional or
  remembered.
- **Block the syntactic, signal the semantic.** The syntactic gate
  exits non-zero and stops the commit, because the judgment is
  objective. The semantic review never blocks — its verdict comes
  from a probabilistic step reading documents, and blocking on an
  unreliable verdict would be worse than the problem it solves.
- **Reuse existing rules for the syntactic layer.** The quality and
  security rules already define formatter-per-language, type check,
  secret scan, and the no-bypass norm; the styling and language rules
  already define naming and identifier conventions. The pre-commit
  layer orchestrates these — it does not redefine them.
- **The semantic reviewer is the one deep module.** It is a new skill
  that reads the living source of truth, confronts the diff, and
  emits structured divergence findings (convention-invented,
  domain-fact-unsupported). It registers in the `quality`
  bundle (beside the other `audit-*` skills) so no skill ships
  loose; the deterministic syntactic layer is already provided by
  the `guardrails` bundle and is adopted rather than rebuilt.
- **Grounding sources, in order:** domain context document, decisions
  of record, knowledge base, and any module-scoped context. Absence
  of a top-level domain context document is tolerated — the reviewer
  degrades to the decisions of record and knowledge base.
- **Distribution.** The pre-commit trigger, the session-stop trigger,
  and the semantic-grounding skill are distributable through the
  bundle so consumer repos inherit them by enabling one entry.

## Testing Decisions

- **Semantic reviewer — unit + scenario tests (the deep module).**
  Tests assert that an invented convention not present in the source
  of truth produces a divergence finding, that an unsupported domain
  claim produces a finding, and that fully grounded output produces
  none. This is the module whose correctness carries the feature, so
  it gets the coverage.
- **Pre-commit gate — integration test.** One test per blocking
  concern (unformatted, type error, secret) asserting a non-zero exit
  and an actionable message; one test asserting clean code passes.
- **Styling wiring — no new tests.** It delegates to the existing
  formatter and existing rules, which are already covered upstream.

## Out of Scope

- Any CI, server-side, or pull-request gate — the layer is local only.
- Blocking on semantic hallucination — explicitly signal-only.
- Creating the source of truth — it already exists in the consumer
  repo via Octopus inputs.
- Syntactic hallucination (non-existent APIs, missing files) and the
  "claimed done without running" failure mode — out of focus for this
  iteration; the chosen scope is invented conventions and false
  domain facts.

## Further Notes

Open questions the implementing agent should resolve before starting:

- The target consumer repo's stack, which determines the exact
  formatter, type-check, and secret-scan commands wired into the
  pre-commit layer.
- Where the session-stop signal is delivered — session output, a
  reviewable proposals queue, or both.
- Whether the semantic review runs on every task stop or only when the
  diff touches domain-relevant files, to control noise.
- Whether RM-088 should be appended to the roadmap as part of this
  work or tracked separately.
