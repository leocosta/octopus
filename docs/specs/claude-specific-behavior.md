## Problem Statement

The `agents/claude/CLAUDE.md` template has a `## Claude-Specific Behavior` section that is completely empty — just a TODO comment. According to Boris Cherny, this is the most impactful section of CLAUDE.md: it is where the meta-instructions that shape Claude's behavior throughout the entire session live.

The 3 practices most recommended by Boris are absent from the template:
1. Update CLAUDE.md after every correction ("Compounding Engineering")
2. Use plan mode for complex tasks before implementing
3. Run `/simplify` after code changes

Teams using Octopus do not receive these instructions automatically.

## Goals

1. Fill in the `## Claude-Specific Behavior` section in the `agents/claude/CLAUDE.md` template with the 3 practices above
2. Keep the section customizable: add a guide comment so teams can add their own instructions
3. Do not modify any other section of the template

## Non-Goals

- Do not make the section configurable via the manifest (that would be a future RM-NNN)
- Do not include instructions already covered by the project's `rules/`
- Do not add tool-specific instructions (e.g. references to Notion, Linear)

## Design

### Overview

Direct edit of the `agents/claude/CLAUDE.md` file. Replace the TODO comment block with the final content.

### Detailed Design

**Current section content** (final lines of the template):

```markdown
## Claude-Specific Behavior

<!-- TODO: Add Claude Code specific instructions -->
<!-- Examples: tool usage preferences, confirmation rules, language/tone -->
```

**New section content**:

```markdown
## Claude-Specific Behavior

### After Every Correction

When you fix a mistake or learn something new about how this project should work,
end your response with: "Update your CLAUDE.md so you don't make that mistake again."
Then update the CLAUDE.md immediately. This is how the team's guidelines improve over time.

### Plan Before Complex Tasks

For any non-trivial task (new feature, refactor, architectural change), enter plan mode
first. Present the plan and wait for approval before writing code.

Trigger plan mode explicitly when:
- The task touches more than 2 files
- You're unsure about the approach
- The user says "think about this" or "what's the best way to..."

### Code Quality After Changes

After completing code changes, append `/simplify` to review the modified code for
reuse, quality, and efficiency before declaring the task done.

---

<!-- Add project-specific Claude instructions below this line -->
<!-- Examples: preferred libraries, confirmation rules, response language -->
```

### Migration / Backward Compatibility

- The template is generated by `setup.sh` on every run. Projects that have already run setup will see the change on the next `setup.sh` execution — the generated CLAUDE.md will be updated.
- Projects that have manually customized their `## Claude-Specific Behavior` section will need to re-apply their customizations after the next setup run. This is expected behavior (setup is idempotent by design).
