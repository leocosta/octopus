## Gaps Identified

### A — Per-task log files (RM-057)

Logs live at `.octopus/logs/<role>.log` and are **overwritten** on every new `launch()` call. Running three tasks for `tech-writer` means only the last task's log is accessible — the previous two are gone. `launch_resume()` appends, but a new `launch()` starts fresh. This is a silent data loss: the TUI shows ✓ for completed tasks and clicking them loads the current log file, but if the role has since been re-used, the log belongs to a different task.

### B — Cancel queued task from TUI (RM-058)

There is no way to remove a task with `status == "queued"` from within `octopus control`. `Ctrl+D` only cleans `done` and `failed` tasks. If a user enqueues the wrong task, the only recourse is to manually delete the JSON file from `.octopus/queue/`. A `x` or `Del` keybind on a selected queued item would fix this.

### C — Retry failed task from TUI (RM-059)

When a task fails (`✗`), `octopus ask` suggests `octopus ask <role> "task" --retry` in the terminal output — but there is no equivalent action in the TUI. The user must leave `octopus control`, run the CLI command, and return. A `[R]etry` keybind (distinct from `[r]` reply) that re-enqueues the failed task would close the loop.

### D — Notification on agent completion (RM-060)

There is no external signal when an agent finishes or fails. For the common "dispatch and go do something else" workflow, the user must keep `octopus control` visible or poll it manually. A terminal bell (`\a`) on task done/failed — and optionally `notify-send` / `osascript` for a desktop notification — would make the TUI usable in the background.

### E — `octopus ask --reply` (RM-061)

`octopus ask` prints the session file path at the end (`session: .octopus/sessions/<role>.session`), but there is no `octopus ask --reply <role> "text"` subcommand to continue the session from the CLI. Replying without the TUI currently requires calling `claude --resume <session_id> --print "text"` manually. This breaks the `octopus ask` abstraction for iterative use from scripts or other tooling.

### F — Model override in TUI command bar (RM-062)

The model is hardcoded to `claude-sonnet-4-6` everywhere the TUI dispatches a task (`on_input_submitted`, `_on_schedule_fire`). There is no way to say `@tech-writer: write the spec --model opus` inline. A `--model <name>` flag parsed from the command bar text (before routing to the skill matcher) would allow per-task model selection without editing the manifest.

### G — Daemon mode (RM-063)

`octopus control` must stay open for the queue to process. Closing the TUI leaves agents running but stops new tasks from being dispatched and sessions from being tracked. A `--daemon` flag (or `octopus control start/stop`) would run the dispatch loop and session tracker as a background process, making the queue usable in CI, long overnight runs, or when you want to close the terminal window.
