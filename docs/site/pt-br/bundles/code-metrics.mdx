`code-metrics` é o irmão de **medição** do
[`quality`](/octopus/bundles/quality/) (o gate bloqueante) e do
[`quality`](/octopus/bundles/quality/) (a revisão advisory).
Enquanto esses dois raciocinam sobre um diff *semanticamente*, este reporta
**números**: como seu branch move cobertura de testes, complexidade ciclomática,
tamanho de módulo e ciclos no grafo de dependências — medidos da mesma forma para
toda mudança, por todo autor.

## Por que isso existe

Os números de qualidade de código degradam silenciosamente, e quem introduz a
regressão é o último a ver. Uma função que cresce de ciclomática 6 para 14, a
cobertura que cai dois pontos, um novo ciclo de import — nada disso se anuncia na
review. Quando um dashboard finalmente mostra a tendência, a mudança já tem
semanas e o contexto se perdeu.

`code-metrics` move essa leitura para a **esquerda**, para o momento em que o
autor ainda pode agir: você recebe o delta localmente, *antes* de abrir o PR. E
precisa ser quase de graça, ou não vai ser rodado — então os números são sempre
computados pelo tooling do seu próprio stack (sem modelo, ≈0 tokens), e um modelo
de baixo custo só é acionado **quando** uma métrica de fato cruza um threshold,
para explicar a regressão e apontar a causa.

## O que inclui

```yaml
skills:
  - code-metrics   # a leitura /octopus:code-metrics + o template da Action escritora
```

O bundle carrega a skill de orquestração e um template de GitHub Action que você
instala por repo. Os **adapters** por-stack — o código que roda o tooling real —
viajam dentro dos stack profiles, não aqui: escolha
[`stack-csharp`](/octopus/bundles/) ou `stack-typescript` e o adapter
correspondente vem junto.

[Source: `bundles/code-metrics.yml`](https://github.com/leocosta/octopus/blob/main/bundles/code-metrics.yml)

## O delta duplo

A leitura reporta **dois** números por métrica, porque "acompanhar ao longo do
tempo" e "o que *este* PR fez" são perguntas diferentes:

- **vs. baseline** — seu branch contra o baseline do último merge na main. É a
  âncora de tendência: a base está drifting pra cima ou pra baixo release a
  release.
- **vs. main** — seu branch contra seu `main` HEAD local. É *sua* contribuição
  isolada: a parte pela qual você de fato responde neste PR.

Colapsar os dois num número só esconde qual metade se moveu. Mantê-los separados
faz um "vs. main" verde te dizer que sua mudança está limpa mesmo quando a
tendência de longo prazo ainda está vermelha por dívida antiga.

## Thresholds — ratchet por padrão

Por padrão **não há config nem alvo absoluto**: cada métrica simplesmente *não
pode regredir* contra o baseline. Um repo legado nunca começa "vermelho" pela
dívida que já carrega — só sinaliza quando uma mudança piora algo.

Quando um time quer puxar o repo pra cima, ele soma alvos absolutos por cima, no
`.octopus.yml`:

```yaml
code_metrics:
  coverage:
    min: 80
  complexity:
    max: 10            # ciclomática, por função
  module_size:
    max: 400           # linhas por módulo
  dependencies:
    cycles_allowed: 0
```

Um campo não definido continua no ratchet. Campos definidos resolvem por camada
com a precedência padrão — um standard compartilhado do time, seus defaults
pessoais e a config commitada do repo, com o **repo vencendo** (um contrato de
qualidade commitado não deve ser relaxado em silêncio pela máquina de um
desenvolvedor).

## O store do baseline

A tendência precisa vir de algum lugar sempre fresco e que nunca conflite com um
branch de PR. O `code-metrics` o mantém num git ref dedicado e não-protegido
(`octopus/code-metrics`), escrito por uma única Action que reage a um push na
`main`. Como a main é serializada pelo squash-merge, há exatamente um escritor —
sem conflito por construção — e a Action só *escreve o ref*, nunca dá push de
volta na `main` ou num branch de release. Sua leitura local apenas faz o fetch.

## O fluxo que isso habilita

1. Você termina uma mudança e, antes de abrir o PR, roda `/octopus:code-metrics`.
2. O adapter do seu stack computa as quatro métricas no seu branch; a leitura faz
   fetch do ref de baseline e imprime o delta duplo. Nenhum modelo entra.
3. Se nada cruzou um threshold, essa é a interação inteira — custo quase zero.
4. Se uma métrica rompeu, e só então, um modelo de baixo custo lê a mudança
   ofensora e explica *por que* ela se moveu e o que fazer.
5. Depois que o PR faz squash-merge, a Action recomputa no commit mergeado e
   avança o baseline para todos.

## O que ele deliberadamente não é

- **Não é um gate bloqueante.** É um sinal para o autor, nunca uma parede de CI.
  A camada bloqueante é o [`quality`](/octopus/bundles/quality/).
- **Não é atribuição por autor.** Todo PR é medido de forma idêntica — não há
  score "humano vs. assistente".
- **Não é mutation testing nem julgamento de layering** — a métrica de
  dependências detecta ciclos, não dá veredito sobre camadas.
