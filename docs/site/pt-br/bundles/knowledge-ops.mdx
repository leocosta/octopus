`knowledge-ops` contém as três skills que mantêm uma base de conhecimento viva
e útil ao longo do tempo. É um subconjunto focado do
[`quality`](/octopus/bundles/quality/) para times que têm uma árvore `knowledge/`
estabelecida e querem operações dedicadas sobre ela — sem necessariamente adotar
toda a superfície de auditoria.

## Por que isso existe

Uma base de conhecimento se degrada silenciosamente. Entradas ficam obsoletas à
medida que o codebase evolui. Conexões cross-domain que ajudariam o próximo
implementador nunca são registradas. Contradições se acumulam entre módulos
escritos com meses de diferença. E quando um novo colega (ou um agent) começa,
não há nenhum sinal sobre qual conhecimento é fresco e qual é arqueologia.

Essas três skills atacam essa degradação nas camadas de manutenção, síntese e
briefing, respectivamente. Não são gates de auditoria — são rotinas operacionais
que mantêm o knowledge loop saudável.

## O que inclui

```yaml
skills:
  - knowledge-hygiene    # obsolescência, órfãos, links quebrados, archive drift
  - knowledge-synthesize # conexões cross-node, contexto esquecido, contradições
  - knowledge-briefing   # o que mudou, o que precisa de atenção, digest proativo
roles: []
```

[Fonte: `bundles/knowledge-ops.yml`](https://github.com/leocosta/octopus/blob/main/bundles/knowledge-ops.yml)

### Por que cada uma está aqui

- **`knowledge-hygiene`** — varre a árvore de conhecimento em busca de entradas
  obsoletas (fatos que contradizem o código atual), nós órfãos (sem links
  entrando ou saindo), referências quebradas e archive drift (entradas promovidas
  a status canônico que não são mais precisas). Produz uma lista priorizada de
  itens que precisam de revisão ou remoção.
- **`knowledge-synthesize`** — lê a árvore de conhecimento em busca de conexões
  que nunca foram registradas: um fato em `knowledge/payments` que esclarece um
  padrão em `knowledge/auth`, uma entrada esquecida mas relevante que o
  implementador atual gostaria de saber, contradições entre dois nós que
  divergiram. Surfaça essas conexões como adições ou correções propostas.
- **`knowledge-briefing`** — gera um digest proativo do que mudou na base de
  conhecimento desde um determinado ponto (última sessão, última semana, último
  release) e sinaliza entradas que precisam de uma decisão humana. Projetado para
  rodar no início de uma sessão ou sprint para orientar o agent e o time.

### Por que algumas coisas não estão aqui

- **`audit-grounding`** — também lê a árvore de conhecimento, mas faz isso como
  uma verificação de divergência signal-only atrelada ao diff de código. Pertence
  ao [`quality-signals`](/octopus/bundles/quality-signals/) porque seu trigger e
  escopo são centrados no diff, não na árvore de conhecimento.
- **`doc-lifecycle`, `doc-adr`** — essas produzem conhecimento; as três deste
  bundle o consomem e mantêm. Os produtores vivem no `starter` e no `docs`.

## Workflow que isso habilita

Essas skills mapeiam para três ritmos distintos:

- **Manutenção sob demanda** — rode `/octopus:knowledge-hygiene` depois que uma
  feature grande for mergeada para capturar entradas que o novo código tornou
  obsoletas. Recebe um path (padrão: `knowledge/`) e produz uma lista de fixes
  priorizados.
- **Síntese periódica** — rode `/octopus:knowledge-synthesize` trimestralmente
  ou após mergear uma feature cross-domain. A skill propõe novas conexões e
  surfaça contradições; você decide quais registrar.
- **Início de sessão ou sprint** — rode `/octopus:knowledge-briefing` para se
  orientar (ou para orientar uma sessão de agent nova) antes de pegar uma tarefa.
  A skill resume o que mudou e sinaliza entradas que precisam de uma decisão
  antes que o trabalho comece.

## Compõe com

- **[starter](/octopus/bundles/starter/)** — provê `doc-adr` e `doc-lifecycle`,
  que produzem as entradas de conhecimento que este bundle mantém.
- **[docs](/octopus/bundles/docs/)** — para o ciclo de vida completo de
  documentação (RFC → Spec → ADR → Knowledge → Changelog) que alimenta a
  árvore de conhecimento.
- **[quality](/octopus/bundles/quality/)** — `knowledge-ops` é um subconjunto do
  `quality`; o bundle completo inclui essas skills junto com os gates de
  auditoria.
