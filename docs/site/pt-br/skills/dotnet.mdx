`dotnet` dá a um agente trabalhando numa base ASP.NET Core o julgamento
arquitetural de um engenheiro .NET experiente — os padrões, convenções e árvores
de decisão "pra onde isso deveria ir?" que fazem a diferença entre código que
compila e código que se encaixa na plataforma.

## O que resolve

.NET oferece várias formas válidas de fazer a mesma coisa — Minimal APIs ou
controllers, esta camada ou aquela — e a escolha certa depende do tamanho do
projeto, do time e das convenções. Um agente generalista pega uma
arbitrariamente; o resultado roda mas briga com a base. Esta skill codifica a
escolha idiomática em cada bifurcação pra que o código pareça pertencer.

## Como resolve

Carrega os padrões e árvores de decisão pro trabalho backend em .NET:

- **Design de API** — quando preferir Minimal APIs (microsserviços, CRUD enxuto,
  menos cerimônia) versus controllers (convenções MVC estabelecidas, filtros e
  model binding complexos), com padrões idiomáticos pra cada um.
- **Arquitetura** — como estruturar um serviço novo e dividi-lo em camadas,
  seguindo as convenções que um projeto ASP.NET Core espera.
- **Detecção de stack** — reconhece um projeto .NET pelos seus `*.csproj` /
  `*.sln`, `Program.cs` / `Startup.cs`, e `appsettings.json`, então ela atua nas
  bases certas.

## Quando usar

Ao implementar ou refatorar uma API ASP.NET Core, projetar um serviço .NET novo,
escolher entre padrões no ecossistema .NET, ou revisar código .NET por qualidade e
aderência às boas práticas.
