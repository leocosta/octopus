Skills são como o Octopus diz ao agent *o que fazer* e *como fazer
com disciplina*. Cada skill é um pequeno arquivo Markdown com
frontmatter que dispara o engajamento conforme a intenção do
usuário — uma descrição que diz "use quando X", `triggers:`
opcionais para auto-engajamento em padrões específicos de arquivo,
e um corpo que conduz o agent pela sequência correta de passos.

## Por que skills, e não só rules

Rules descrevem *como é um bom código* (coding style, security,
patterns) — são carregadas em toda conversa como contexto
permanente. Skills descrevem *como executar uma tarefa* — são
carregadas sob demanda quando a intenção do usuário casa com a
descrição da skill.

Essa divisão mantém o contexto ativo enxuto: uma rule sobre
Conventional Commits está sempre carregada, mas toda a disciplina
de PR-feedback só carrega quando você está de fato recebendo
feedback em um PR. Skills são a camada "engaja quando preciso".

## Granularidade

Conforme a rule do docs-site, apenas skills com decisões de design
não-triviais ganham uma página de detalhe. Padrões específicos de
stack, wrappers finos e utilitários de uma única função ficam
listados nas tabelas de visão geral, sem página própria.

## Audits

Gates de auditoria pré-merge. Cada um roda read-only e produz um
relatório por severidade (`⛔ block` / `⚠ warn` / `ℹ info`). O
compositor `audit-all` executa os relevantes em paralelo.

| Skill | Audita | Detalhe |
|---|---|---|
| `audit-all` | Compositor para os quatro audits abaixo — descoberta de arquivos compartilhada, execução paralela, relatório consolidado. | [→](./audit-all/) |
| `audit-security` | Segredos hardcoded, riscos de injection, lacunas de auth em rotas públicas, CVEs de dependências. | [→](./audit-security/) |
| `audit-money` | Tipos monetários, arredondamento, testes com centavos não-redondos, consistência de env-vars, idempotência de pagamentos, verificação de assinatura de webhooks, acoplamento de fee-disclosure. | [→](./audit-money/) |
| `audit-tenant` | Enforcement de escopo de dados multi-tenant — filtros de query, raw SQL, configs de novas entities, ownership em controllers, endpoints administrativos. | [→](./audit-tenant/) |
| `audit-config` | Atualidade da configuração do agent — premissas específicas de modelo, datas obsoletas, skills fantasmas, paths depreciados. | [→](./audit-config/) |
| `audit-grounding` | Divergência signal-only da fonte de verdade — convenções inventadas, fatos de domínio sem suporte que o formatter não enxerga. | [→](./audit-grounding/) |
| `audit-fleet` | Adoção + drift cross-repo — config de cada repo vs o alvo declarado (baseline + perfil de stack + tier de adoção) do fleet.yml. Signal-only; alimenta o fleet-bootstrap. | [→](./audit-fleet/) |
| `review-contracts` | Drift de contrato entre frontend/backend — endpoints, DTOs, enums, status codes, regras de auth, parâmetros. | [→](./review-contracts/) |

## Ciclo de vida de docs

A cadeia completa RFC → Spec → ADR → Knowledge. Cada etapa é uma
skill que o comando correspondente invoca.

| Skill | Saída | Detalhe |
|---|---|---|
| `doc-lifecycle` | Roteia entre RFC / Spec / ADR / Knowledge conforme escopo e risco — a skill de entrada para trabalho de documentação. | [→](./doc-lifecycle/) |
| `doc-adr` | Um novo Architecture Decision Record em `docs/adr/`. | [→](./doc-adr/) |
| `doc-plan` | Um plan de implementação em estilo TDD, fatiado em pedaços pequenos, a partir de uma spec aprovada. | [→](./doc-plan/) |
| `doc-subcontext` | `CLAUDE.md` por módulo para monorepos — herda do parent. | [→](./doc-subcontext/) |
| `doc-align` | Reconcilia o drift entre código e docs. | — |
| `doc-prd` | PRD publicado no issue tracker do projeto. | — |
