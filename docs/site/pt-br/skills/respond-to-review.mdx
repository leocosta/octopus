`respond-to-review` é a disciplina pra lidar bem com feedback de PR: leve cada
comentário a sério, mas não mude código só pra parecer concordante. Roda o loop
inteiro — de julgar a crítica a responder e resolver threads — ponta a ponta.

## O que resolve

O modo de falha de um agente (ou pessoa) ansioso sob review é a **mudança
performática**: editar o código pra satisfazer um comentário sem checar se o
comentário está certo, então uma nota vaga ou equivocada é implementada e o diff
deriva. A falha oposta é igualmente comum — descartar feedback fundamentado como
preferência. Boa resposta a review exige julgamento, e esta skill o codifica em vez
de deixá-lo pro reflexo.

## Como resolve

A disciplina é um conjunto de regras permanentes:

- **Verificar a crítica** antes de agir — ela está correta, contra este código?
- **Pedir evidência** em comentários genéricos ("isso está lento" → lento como, onde?).
- **Separar feedback fundamentado de preferência**, e pesá-los diferente.
- **Nunca fazer uma mudança performática** — se um comentário não se sustenta,
  diga isso com raciocínio em vez de editar pra apaziguar.
- **Pedir esclarecimento** quando um comentário é ambíguo, em vez de chutar.

Uma vez que os fixes são acordados e aplicados, roda o **loop pós-fix num turno** —
commit, push condicional, replies inline, resolver threads — então é ponta-a-ponta
e não precisa de um passo `/octopus:pr-comments` separado.

## Quando usar

Toda vez que você está agindo sobre feedback de review de PR — é ativa por padrão
nesse loop. As mudanças de código em si são então feitas via `implement` ou
`debugging`.
