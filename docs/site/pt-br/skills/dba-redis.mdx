`dba-redis` é o catálogo de heurísticas do Redis que dá à role `dba` o olho de um
especialista de cache-e-datastore sobre uma mudança. O Redis é simples de começar
e fácil de usar errado — e porque vive em memória e no caminho quente, seus erros
falham alto. Todo achado vem com um comando Redis ou fix de config.

## O que resolve

Problemas de Redis são operacionais, não só de correção: uma chave sem TTL que
vaza memória até o nó despejar dado vivo, uma única "big key" que bloqueia o event
loop em todo acesso, uma hot key que prende um shard, uma config de persistência
que troca em silêncio a durabilidade que você assumiu ter. Um generalista não pesa
esses trade-offs. Esta skill os nomeia antes de o cache virar o incidente.

## Como resolve

Roda um catálogo de heurísticas específicas do Redis:

- **design de chave & disciplina de TTL** — nomeação, e um expiry em tudo que não
  deve viver pra sempre;
- **riscos de hot / big key** — padrões de acesso e tamanhos de valor que bloqueiam
  o loop ou fazem hot-spot num shard;
- **trade-offs de persistência** — RDB vs. AOF contra a durabilidade que você de
  fato precisa;
- **pub/sub vs. streams**, **uso de pipeline**, **política de eviction**, **riscos
  operacionais** — o primitivo certo, menos round-trips, uma política `maxmemory`
  sã, e os footguns que times de ops batem.

Cada achado carrega uma **severidade** e um **comando Redis ou fix de config**
concreto.

## Quando usar

Em qualquer mudança que usa Redis — um cache novo, uma fila, um rate limiter, um
design de pub/sub ou streams. É o catálogo Redis da role `dba`, completando o
conjunto com `dba-mssql`, `dba-postgres` e `dba-mongodb`.
