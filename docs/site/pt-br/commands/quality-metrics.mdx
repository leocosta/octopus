`/octopus:quality-metrics` roda uma leitura local de como seu branch move quatro
métricas determinísticas de qualidade de código, antes de você abrir o PR — e
continua quase de graça a menos que algo de fato regrida.

## O que resolve

Quem introduz uma regressão de qualidade costuma ser o último a descobrir: a
review deixa passar uma complexidade crescente, uma queda de dois pontos na
cobertura ou um novo ciclo de import, e um dashboard só mostra a tendência muito
depois do merge. Este comando move a leitura para o momento em que você ainda
pode corrigir barato, e a mantém barata computando os números com o tooling do
seu próprio stack em vez de um modelo.

## Como funciona

Um adapter por-stack — escolhido a partir do stack detectado — roda o tooling
real (cobertura, complexidade ciclomática, tamanho de módulo, ciclos de
dependência) e normaliza a saída para um contrato compartilhado. O comando faz
fetch do baseline por-merge no ref `octopus/quality-metrics` e imprime um **delta
duplo**: seu branch vs. esse baseline (a tendência) e vs. seu `main` local (o
impacto deste PR).

Os thresholds são ratchet por padrão (sem regressão vs. baseline), com alvos
absolutos opcionais no `.octopus.yml`. A rodada numérica nunca chama um modelo;
só uma quebra de threshold aciona um modelo de baixo custo para explicar a
regressão e sugerir um fix.

## Uso & parâmetros

```
octopus quality-metrics [--stack <id>] [--against <ref>]
```

- `--stack <id>` — força um adapter de stack (`csharp`, `typescript`) em vez de
  auto-detectar a partir do repo.
- `--against <ref>` — o ref local usado como comparador "vs. main"; padrão
  `main`.
