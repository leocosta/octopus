`quality-audits` is the subset of [`quality`](/octopus/bundles/quality/) that
draws a hard line between **blocking gates** and advisory signals. It installs
only the skills and roles that can block a merge — nothing that is
signal-only by design. If you want the advisory layer too, pick the full
`quality` bundle or add [`quality-signals`](/octopus/bundles/quality-signals/)
alongside this one.

## Why this exists

`quality` is the right default for production repos. But some teams arrive at
audits incrementally: they want the blocking gates today and will layer in
signals later once the team is comfortable with the workflow. Other teams use
their own advisory tooling and only need Octopus for the blocking layer.
`quality-audits` is that precise scope — no more.

## What it includes

```yaml
skills:
  - audit-all         # parallel composer: audit-security, audit-money, audit-tenant
  - audit-contracts   # cross-stack contract drift (DTO/endpoint/enum/status-code)
roles:
  - architect         # classifies findings: BLOCKING / ADVISORY / QUESTION
  - security          # threat-modeling pass on auth and secret diffs
```

`audit-all` installs `audit-security`, `audit-money`, and `audit-tenant` as
transitive dependencies.

[Source: `bundles/quality-audits.yml`](https://github.com/leocosta/octopus/blob/main/bundles/quality-audits.yml)

### Why each one is in

- **`audit-all`** — the entry point. Runs the four domain audits in parallel
  against a branch or PR diff, producing a consolidated severity-tiered report
  (`⛔ block`, `⚠ warn`, `ℹ info`) with cross-audit hotspots.
- **`audit-contracts`** — catches drift between API producers and consumers that
  the per-file domain audits don't see: mismatched DTOs, renamed enum values,
  removed endpoints, changed status codes across the stack boundary.
- **`architect` role** — audit findings need human-grade classification. This
  role walks a consolidated report and labels each finding as blocking, advisory,
  or needs-discussion, so the review conversation is structured, not a stream of
  raw output.
- **`security` role** — adds a threat-modeling pass on top of the `audit-security`
  checklist: attack surface, auth flows, secret handling. The `codereview` router
  dispatches it automatically on diffs that touch auth or secrets, dual-gated
  with `architect`.

### Why not included

- **`audit-grounding`, `audit-verification`, `audit-style`, `audit-config`,
  `refactor-deepen`** — those are signal-only and belong in
  [`quality-signals`](/octopus/bundles/quality-signals/). They never block a
  merge; mixing them here would dilute the blocking-gate contract.
- **`knowledge-hygiene`, `knowledge-synthesize`, `knowledge-briefing`** — those
  are knowledge-loop operations, in [`knowledge-ops`](/octopus/bundles/knowledge-ops/).

## Workflow this enables

1. You finish a feature and push your branch.
2. You run `/octopus:audit-all` before opening the PR. The skill dispatches
   the four audits in parallel against the diff and consolidates findings.
3. If the change touches the API boundary, `audit-contracts` inspects the diff
   for DTO / endpoint / enum / status-code drift across the stack.
4. Any finding labelled `⛔ block` must be resolved before merge. The `architect`
   role walks the report and reclassifies edge cases; the `security` role adds a
   threat-model pass on auth or secret changes.

## Composes with

- **[starter](/octopus/bundles/starter/)** — required foundation (provides the
  implementation workflow that produces the code being audited).
- **[quality-signals](/octopus/bundles/quality-signals/)** — the advisory complement
  to this bundle; together they cover the full `quality` surface.
- **[quality](/octopus/bundles/quality/)** — if you want the complete surface in a
  single install, use `quality` instead of combining `quality-audits` and
  `quality-signals` manually.
