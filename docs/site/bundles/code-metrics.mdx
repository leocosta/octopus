`code-metrics` is the **measurement** bundle: it reports **numbers** on how
your branch moves test coverage, cyclomatic complexity, module size, and
dependency-graph cycles — measured the same way for every change, by every
author. Where [`quality`](/octopus/bundles/quality/) reasons about a diff
semantically (audits, advisory signals), this bundle answers the quantitative
question.

## Why this exists

Code-quality numbers degrade quietly, and the person who introduces the
regression is the last to see it. A function that grows from cyclomatic 6 to 14,
coverage that slips two points, a new import cycle — none of these announce
themselves in review. By the time a dashboard surfaces the trend, the change is
weeks old and the context is gone.

`code-metrics` moves that read **left**, to the moment the author can still
act on it: you get the delta locally, *before* you open the PR. And it has to be
near-free, or it won't get run — so the numbers are always computed by your
stack's own tooling (no model, ≈0 tokens), and a low-cost model is invoked
**only** when a metric actually crosses a threshold, to explain the regression
and point at the cause.

## What it includes

```yaml
skills:
  - code-metrics   # the /octopus:code-metrics read + the writer-Action template
```

The bundle carries the orchestration skill and a GitHub Action template you
install per repo. The per-stack **adapters** — the code that runs the real
tooling — ship inside the stack profiles, not here: pick
[`stack-csharp`](/octopus/bundles/) or `stack-typescript` and the matching
adapter comes with it.

[Source: `bundles/code-metrics.yml`](https://github.com/leocosta/octopus/blob/main/bundles/code-metrics.yml)

## The dual delta

The read reports **two** numbers per metric, because "track over time" and
"what did *this* PR do" are different questions:

- **vs. baseline** — your branch against the last-merged-to-main baseline. This
  is the trend anchor: is the codebase drifting up or down release over release.
- **vs. main** — your branch against your local `main` HEAD. This is *your*
  contribution in isolation: the part you are actually responsible for in this PR.

Collapsing them into one figure hides which half moved. Keeping them apart means
a green "vs. main" tells you your change is clean even when the long-term trend
is still red from older debt.

## Thresholds — ratchet by default

Out of the box there is **no config and no absolute target**: every metric
simply *may not regress* against the baseline. A legacy repo never starts "red"
for debt it already carries — it only flags when a change makes something worse.

When a team wants to pull a repo upward, it layers absolute targets on top, in
`.octopus.yml`:

```yaml
code_metrics:
  coverage:
    min: 80
  complexity:
    max: 10            # cyclomatic, per function
  module_size:
    max: 400           # lines per module
  dependencies:
    cycles_allowed: 0
```

A field left unset stays on the ratchet. Set fields resolve per layer with the
standard precedence — a shared team standard, your personal defaults, and the
repo's committed config, with the **repo winning** (a committed quality contract
should not be silently relaxed by one developer's machine).

## The baseline store

The trend has to come from somewhere that is always fresh and never conflicts
with a PR branch. `code-metrics` keeps it on a dedicated, non-protected git
ref (`octopus/code-metrics`), written by a single Action that reacts to a push
to `main`. Because main is serialized by squash-merge, there is exactly one
writer — conflict-free by construction — and the Action only ever *writes the
ref*, never pushes back to `main` or a release branch. Your local read just
fetches it.

## Workflow this enables

1. You finish a change and, before opening the PR, run `/octopus:code-metrics`.
2. The adapter for your stack computes the four metrics on your branch; the read
   fetches the baseline ref and prints the dual delta. No model is involved.
3. If nothing crossed a threshold, that's the whole interaction — near-zero cost.
4. If a metric breached, and only then, a low-cost model reads the offending
   change and explains *why* it moved and what to do about it.
5. After the PR squash-merges, the Action recomputes on the merged commit and
   advances the baseline for everyone.

## What it deliberately isn't

- **Not a blocking gate.** It is a signal to the author, never a CI wall. The
  blocking layer is [`quality`](/octopus/bundles/quality/).
- **Not author attribution.** Every PR is measured identically — there is no
  "human vs. assistant" scoring.
- **Not mutation testing or architecture-layering judgment** — the dependency
  metric detects cycles, it does not rule on layering.

## Composes with

- **[starter](/octopus/bundles/starter/)** — the foundation that produces the
  code being measured.
- **[quality](/octopus/bundles/quality/)** — the gate and signal axes; `code-metrics`
  is the separate, quantitative measurement axis alongside them.
- **a stack profile** (`stack-csharp`, `stack-typescript`) — required, since the
  numeric adapter for your language ships there.
