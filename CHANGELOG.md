# Changelog

All notable changes to this project will be documented in this file.

## [1.85.1] - 2026-06-09

🐛 **`octopus setup`/`update` no longer aborts on a renamed or removed bundle.** A `.octopus.yml` written by an older Octopus — e.g. listing `knowledge-ops`, renamed to `knowledge` back in v1.83.0 — made the upgrade die with `error: knowledge-ops`, because `_load_bundle` returned non-zero on a missing bundle file and `setup.sh` runs under `set -e`. Config from an earlier version could brick the very upgrade meant to fix it (and, as a knock-on, the aborted setup never regenerated agent configs — so newly-shipped surfaces like the Copilot prompt files silently failed to appear). Loading is now resilient: a shared rename alias map (`knowledge-ops` → `knowledge`; `code-metrics`/`quality-audits`/`quality-signals`/`quality-metrics` → `quality`) resolves known renames in **both** the delivery path and the interactive picker, and any other unknown bundle is warned-and-skipped rather than fatal. 🧪 New `test_bundle_resilience` plus an updated `test_bundles` lock the non-fatal contract, alias resolution, and picker parity.

## [1.85.0] - 2026-06-09

✨ **GitHub Copilot now gets the Octopus workflow commands.** Copilot was configured with rules + MCP but never received `/octopus:*` — its manifest is `native_commands: false`, so the command files were skipped. A new capability-gated delivery path renders each `commands/*.md` as a Copilot **prompt file** under `.github/prompts/octopus-<name>.prompt.md` (frontmatter `mode: agent`, `$ARGUMENTS`→`${input}`), so an IDE (VS Code, Visual Studio, JetBrains) exposes `/octopus-pr-open`, `/octopus-release`, etc. The Copilot CLI keeps the existing `octopus <name>` text fallback (it has no prompt-file support yet — github/copilot-cli#618). Generated prompt files are git-ignored, like the other per-agent surfaces. Recorded as **ADR-011** (delivery is gated on a manifest capability, never on the agent name). (#199, RM-156)

✨ **`octopus setup` now lets you choose the agents.** The interactive picker configured bundles, skills, hooks, commands, reviewers, and MCP — but not which assistants to set up, so enabling Copilot meant hand-editing `.octopus.yml`. The picker gained an agent-selection screen (fzf + bash fallback) sourced from `agents/*/manifest.yml`, defaulting to the agents already in your manifest; a new `--agents <csv>` flag covers the non-interactive path. 🧪 New coverage in `test_setup_agents` and `test_prompt_file_commands`. (#199, RM-157)

## [1.84.5] - 2026-06-09

🐛 **`/octopus:pr-review` now signs its aggregated report.** The PR-posting surfaces were inconsistent — only `/octopus:pr-open` signed its output, while `pr-review` posted a substantial review report as a general PR comment with no attribution. The report now ends with the Octopus footer (`---` + `<sub>🐙 generated by Octopus</sub>`) before it's posted. Inline thread replies (`/octopus:pr-comments`, `/octopus:respond-to-review`) deliberately stay **unsigned** — a per-thread footer would clutter the conversation; only general comments and PR bodies are signed. 🧪 A new `test_pr_signature.sh` locks the distinction as a regression guard. (#198)

## [1.84.4] - 2026-06-09

🐛 **`code-metrics` no longer inflates `magic_numbers` on C#/.NET repos** — a real .NET checkout reported ~39 000, almost all false positives. Two root causes, both fixed. First, pruning was shallow: machine-generated C# that lives in the source tree (EF `Migrations/`, `*.Designer.cs`, `*.g.cs`, `*.generated.cs`, `AssemblyInfo.cs`) was counted; it's now excluded from **every** C# metric, not just this one. Second, the counter stripped comments and strings line-by-line, so any multiline construct leaked its digits — the heuristic is rewritten as a stateful, language-aware `awk` preprocessor that blanks block comments (`/* */`), C# verbatim (`@"…"`) and raw (`"""…"""`) strings, and TS template literals across line boundaries, then drops named-constant context: `const`/`readonly`/`enum`/`#define` and enum bodies for all stacks, plus attribute args and auto-property initializers for C# only (a leading `[` stays an array literal in TypeScript). 🧪 Each false-positive scenario is locked by a regression test (suite at PASS=106), with guards keeping genuine literals — indexers and array elements — counted. (#197)

## [1.84.3] - 2026-06-07

🔒 **`install.sh` now verifies on the default GitHub path.** SHA-256 + GPG verification (and `OCTOPUS_REQUIRE_SIGNATURE`) only ran when `OCTOPUS_INSTALL_ENDPOINT` was set — a plain `install.sh --version vX` from GitHub resolved no checksum/signature URL and skipped both **silently**, making `OCTOPUS_REQUIRE_SIGNATURE=1` a no-op on the most common install path. Both URLs now fall back to the GitHub release asset (the `.sha256`/`.tar.gz.asc` the signing pipeline already publishes), and `OCTOPUS_REQUIRE_SIGNATURE` fails closed when no signature URL resolves. Verified against v1.84.2 (key present → valid; absent → fail-closed). See RM-155.

## [1.84.2] - 2026-06-07

🔒 **Release signing key rotated.** The previous signing key's passphrase was lost — it lived only in the CI secret (write-only, unrecoverable), leaving the key **unrevocable** by hand; automated signing kept working, but the maintainer could no longer operate it. Rotated to a fresh RSA-4096 key (`63C35E66917CE4540CD27592C8BA059A0322F3CD`, expires 2028, clean UID): new keypair, updated release secrets, public key published to keys.openpgp.org (email-verified). The `code-metrics` writer-Action pin (`OCTOPUS_VERSION` → v1.84.2, `OCTOPUS_FPR` → the new fingerprint) is bumped together so consumers verify the v1.84.2 tarball — the first release signed by the new key. The old key is **retired, not revoked** (its private half never leaked — it became inaccessible, not public). See RM-154.

## [1.81.0] - 2026-06-05

⚡ The C# **`quality-metrics`** coverage metric is far faster and more meaningful. It now prefers **`dotnet-coverage`** (binary instrumentation) over coverlet's XPlat collector, which can be pathologically slow on large, async-heavy suites — on a real .NET 8 repo a coverage run that never finished in 10 minutes now completes in ~40s, falling back to coverlet only when `dotnet-coverage` is absent. Two optional `quality_metrics.coverage` fields tune what is measured: **`test_filter`** appends a `dotnet test --filter` (e.g. `Category!=Integration`, to measure unit-only coverage and leave slow end-to-end tests as a separate CI gate), and **`settings`** points at a dotnet-coverage settings file to scope the report to production assemblies and exclude generated code such as EF migrations. Every `find`/`lizard` scan in the C# adapter now also prunes `.claude/` agent worktrees and `node_modules/`, so a stale repo copy can neither skew the numbers nor be picked as the test project. 🔒 A security review hardened the new string config: `coverage.test_filter` is allowlisted against the `dotnet test --filter` grammar (mirroring the numeric gate), so an attacker-influenceable `.octopus.yml` cannot inject extra arguments into the command. 🐛 The `octopus setup` picker's **`ESC`-to-go-back** no longer aborts setup under `set -e` (a return-code capture bug). ♻️ Stack and database choices are presented as atomic profiles rather than one-item bundles.

## [1.79.0] - 2026-06-04

✨ This release adds **`quality-metrics`** — a new measurement axis for code quality, sibling to the audit (`quality-audits`) and signal (`quality-signals`) bundles. Where those reason about a diff semantically, this one reports numbers: how a change moves **test coverage, cyclomatic complexity, module size, and dependency-graph cycles**. The author runs `/octopus:quality-metrics` locally before opening a PR and gets a **dual delta** — one figure against the last-merged-to-main baseline (the long-term trend) and one against local `main` (this PR's isolated impact). The numbers are always computed by the stack's own tooling at essentially zero token cost; a low-cost model is invoked **only** when a metric crosses a threshold, to explain the regression and suggest a fix. Thresholds are **ratchet-by-default** — a metric may not regress against the baseline, so a legacy repo never starts "red" — with optional absolute targets layered per metric in `.octopus.yml` (resolved across the workspace → personal → project layers, the repo winning). The per-merge baseline lives on a dedicated, non-protected git ref written by an Action reacting to a push to `main`, so it stays fresh, conflict-free under squash-merge, and never pushes back to a protected branch. v1 ships **C# and TypeScript** adapters, selected by the existing stack detection; the bundle carries the orchestration skill plus the writer-Action template installed per repo. 🐛 A security review of the merge closed an awk code-injection where attacker-influenceable config and baseline values reached the metric arithmetic as program text — values are now validated numeric at the boundary and passed to awk as data, never code. 📝 The published site documents the bundle, skill, and command in English and pt-br.

## [1.77.1] - 2026-06-03

🐛 `octopus update` now propagates changes to the shim itself, not just the cached release. The shim (`bin/octopus`) is copied into `~/.local/bin`, but `command_update` invoked the installer with `--no-shim-setup`, so a change to the shim — like the `version` command added in v1.77.0 — only reached users through a full reinstall, never through `update`; the shim could stay frozen across releases while the cache moved forward. Two complementary layers fix this: `sync_shim` in the shim refreshes the running binary from the target release via an atomic rename (safe to overwrite the executing script — the open inode survives until exit), making every future shim change self-propagating; and because old shims re-fetch `install.sh` fresh from `main` on every update, `heal_stale_shim` in the installer heals an already-installed stale shim on a forward-version move (never a downgrade or pinned-older backfill, never creating a stray shim), closing the gap for the installed base with no manual copy. 🧪 New regression tests lock both paths: `test_shim_sync.sh` (update refreshes a stale shim, second update is a no-op) and `test_shim_heal.sh` (forward heal, no-downgrade, no stray shim, no temp leak).

## [1.77.0] - 2026-06-02

✨ The CLI-surface-hygiene cluster lands its conventional affordances, reference docs, and a real `doctor`. Building on the command registry (v1.76.0), `octopus version` / `--version` now prints the installed version (it used to error with "Unknown command"), `octopus list` emits the command names for scripting, `octopus help <command>` gives a one-line summary, and `octopus completions [bash|zsh|fish]` generates a shell-completion script — while each command keeps its own richer `--help`. 📝 A new bilingual CLI Reference page (EN + pt-br) documents the surface that was accepted but undocumented: the full `setup` flag set (`--no-hooks`, `--no-workflow`, `--bundle`, `--stack`, `--reviewers`), the `release` subcommands, and the `OCTOPUS_*` environment variables. ✨ `octopus doctor` grows from "print version + path" into a read-only health command surfacing the failure classes that actually bite: stale hook paths in `settings.json`, broken cache symlinks, and an available newer release. 🐛 Relatedly, `octopus setup` no longer leaves those stale hook paths behind: `deliver_hooks` deduplicated by hook id with a "skip if present" merge, so after an update the old version-pinned path stayed in `settings.json` (pointing at a deleted cache dir) while the refreshed entry was dropped, erroring at session start. The merge now prunes Octopus-owned hooks and re-adds them at the current path, preserving user-added hooks — so every repo self-heals on its next setup or update.

## [1.76.0] - 2026-06-02

✨ The workflow CLI gains a **command registry**, replacing the "a command exists because a file exists" dispatch. `cli/octopus.sh` used to `source cli/lib/<cmd>.sh` with no allowlist, so every helper or implementation library in `cli/lib/` (`ui`, `audit-map`, `knowledge-*`, `consigliere-lens`) was a silently-accepted command that defined functions and did nothing — and the two hand-maintained help strings drifted from the real set, so `octopus --help` only ever showed the five bootstrap commands. A new declarative registry, `cli/lib/commands.default` (pipe-delimited, mirroring `knowledge-roots.default`), is now the single source of truth: `cli/octopus.sh` generates its command list from it and **rejects any unregistered name** (helper libs now error with "Unknown command" instead of no-op'ing), `help`/`-h`/`--help` is a first-class command, and the `bin/octopus` shim reads the registry so `octopus help` lists the global and workflow commands together — ending the two-layer split. Adding a command stays one step per side: a `cli/lib/<name>.sh` plus one registry line. 🧪 A new `test_cli_registry.sh` locks the guard, the registry-to-lib integrity, and the generated help in place. Keystone of the CLI-surface-hygiene cluster; `version`, per-command `--help`, `list`/`completions`, and a richer `doctor` build on it next.

## [1.74.0] - 2026-06-02

✨ This release adds **`audit-style`** — the design/quality signal the "Quality, Style & Grounding" guardrail set named but never built. The syntactic block (the `guardrails` bundle) and the semantic grounding signal (`audit-grounding`) already shipped; the missing third reader is the one that judges the diff against the team's *opinionated* coding rules. `audit-style` loads `rules/common/exceptions.md` (the custom-exception gate), `patterns.md`, `coding-style.md`, and the active stack rules, then surfaces two failures a formatter and the generic `simplify` cannot see: code that breaks a stated rule (`rule-violation` — a custom exception that fails the gate, a throw where a typed result is called for, a boolean parameter, a magic number, business logic leaking into a repository) and abstraction the rules call a cost (`over-engineering` — premature abstraction, a speculative subtype hierarchy, DRY applied before three occurrences). That over-engineering dimension is what the native `simplify` structurally cannot produce — it optimizes for *less code* and can introduce the very abstraction the rules forbid, so `audit-style` is the reader that knows when **not** to simplify. It mirrors `audit-grounding`'s shape — signal-only, capped at `warn`/`info`, never blocking — but points at the coding rules instead of the domain. It registers in the `quality` bundle and runs through the review flows (`codereview`/`pr-review`/`implement`); unlike its siblings it ships no Stop hook, and it stays out of `audit-all` (a fixed composer of the four domain code audits). Recurring findings reuse the existing continuous-learning loop so a violation the team keeps making is promoted into a rule rather than re-flagged forever. 📝 The public site documents the skill in English and pt-br.

## [1.73.0] - 2026-06-02

✨ The destructive-action guard gains a strict temp-dir carve-out: a clean `rm -rf` confined entirely to `/tmp` or `/var/tmp` is now allowed without the bypass marker, so agents can clean up throwaway scratch (mockups, snapshots) without friction. The carve-out never exempts a command that could reach outside the temp root — because `rm` acts on the live filesystem, the decision rejects shell composition, expansion, and globs, then resolves every target with `realpath` to confirm strict containment, closing the `.`, glob, bare-root, and symlink escapes. It was reviewed adversarially before merge. 🔧 A broad scrub removes Octopus's own internal development markers (roadmap IDs, cluster names, and links/pointers to Octopus's internal ADRs, specs, and RFCs) from the three surfaces a consumer actually sees — the skills, commands, and templates that `octopus setup` copies into a repo, and the published documentation site — while preserving every user-facing ADR/RFC/spec/roadmap feature and the user's own roadmap-ID convention. 🧪 Two regression guards lock this in: the site's `check-docs.sh` now also flags links into Octopus's internal design docs, and a new `test_no_internal_refs.sh` blocks cluster names and internal ADR citations from leaking back into the delivered skills, commands, and templates.

## [1.71.1] - 2026-05-31

🐛 Fixed `octopus setup` silently skipping bundle members that carry inline `# RM-NNN …` annotations. Both hand-rolled YAML parsers in `setup.sh` — `_load_bundle` (reads `bundles/*.yml`) and `parse_octopus_yml` (reads the user-written `.octopus.yml`) — were appending the trailing inline comment to the skill/role name, so setup looked for a directory like `consigliere-bootstrap     # RM-099 — …` and skipped it as "not found". It is not wired into CI yet — it currently reports 48 leaking pages, the sweep worklist, and CI gating lands once that sweep clears.

## [1.0.0] - 2026-04-18

Marks the first stable release, consolidating RM-005 through RM-009 in a single delivery.

✨ The global CLI installer now verifies detached GPG signatures alongside the existing SHA256 check — configurable via `OCTOPUS_GPG_KEYRING`, `OCTOPUS_GPG_IMPORT_KEY`, `OCTOPUS_REQUIRE_SIGNATURE`, and `OCTOPUS_SKIP_SIGNATURE`, closing the remaining supply-chain gap on compromised mirrors. Installer hardening continues with new `--bin-dir` / `--cache-root` flags, `OCTOPUS_INSTALL_ENDPOINT` support for `file://` mirrors, and real SHA256 capture in `metadata.json`.

✨ Setup UX was unified across `install.sh`, the interactive wizard, and `setup.sh`. A new `cli/lib/ui.sh` module provides shared vocabulary that groups per-agent delivery under a single step line. Every wizard prompt now dispatches to the active TUI backend (fzf / whiptail / dialog / bash), and each of the eleven wizard steps gained a contextual explanation plus per-item hints.

📝 Role templates can now declare a `tools:` frontmatter field that is preserved for Claude Code and stripped for every other target. Language rules specs were promoted to Implemented alongside the behavioral detection rule and the project-override mechanism.

⚠️ BREAKING CHANGE: submodule mode is no longer supported. `cli/lib/update.sh`, `commands/update.md`, `tests/test_update.sh`, the submodule branch in `setup.sh` PROJECT_ROOT resolution, `OCTOPUS_CLI_REL`, and every submodule/legacy-shim reference across README, commands, and feature docs were removed. Existing submodule installs must switch to the global CLI (`install.sh`) and re-run `octopus setup`; the manifest is preserved. RM-010 (`octopus migrate` helper) was rejected as a consequence.
