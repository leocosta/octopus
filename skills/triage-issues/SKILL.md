# Issue Triage Protocol

## Overview

Move issues through a small state machine using disciplined
transitions so an AFK agent (or a human) can pick up `ready-for-*`
issues with full context. The skill enforces transparency about AI
involvement and preserves work even when an issue is blocked on the
reporter.

## Categories and States

**Categories**: `bug` (current behaviour is wrong) /
`enhancement` (current behaviour is correct, change requested).

**States**:

| State | Meaning | Exit transitions |
|---|---|---|
| `needs-triage` | Just landed | → any other state |
| `needs-info` | Blocked on reporter | → `ready-for-*` on reply; → `wontfix` on timeout |
| `ready-for-agent` | An AFK agent can implement | → `in-progress` (picked up) |
| `ready-for-human` | Needs human judgement first | → `ready-for-agent` after decision |
| `wontfix` | Rejected — recorded for posterity | terminal |

## Protocol

### Step 1 — Read context

Pull issue body, comments, related ADRs, and `.out-of-scope/` entries.
If the issue echoes an `.out-of-scope/` entry, close as `wontfix`
citing the prior entry — no re-litigation.

### Step 2 — Recommend category + state

Output one sentence: recommended category, recommended state, reason.
Wait for confirmation before applying labels.

### Step 3 — For bugs, reproduce before grilling

Try to reproduce **before** any grilling. Failure to reproduce is a
strong signal for `needs-info`, not for `wontfix`. Reproduction steps
go into a triage comment so the next reader does not redo the work.

### Step 4 — Grill if the issue is unclear

For ambiguous bugs and most enhancements, hand off to `doc-align` for
a grilling sub-loop. Grilling that hits a wall blocked on the
reporter produces a `needs-info` note (see Step 5).

### Step 5 — Preserve `needs-info` work

When transitioning to `needs-info`, write a comment with two
sections:

```
**Established so far:**
- <facts pinned down during grilling>

**Still needed:**
- <specific questions for the reporter>
```

This preserves the grilling investment so the next agent does not
restart the conversation.

### Step 6 — Record rejected enhancements

Closing an `enhancement` as `wontfix` requires a one-paragraph entry
to `.out-of-scope/<slug>.md`: what was proposed, why rejected, cite
to the ADR / spec / CONTEXT.md term that made it out-of-scope. This
is the institutional memory that prevents the same suggestion from
cycling through triage every six months.

### Step 7 — Hand off `ready-for-agent` cleanly

Attach a brief comment listing acceptance criteria, the files /
modules most relevant, and the skills the agent should invoke (e.g.
`test-tdd`, `debug`, `implement`). This brief is durable across
agent invocations.

## The AI Disclaimer

**Every comment generated during triage** carries this trailer:

```
> *This was generated by AI during triage.*
```

Non-negotiable. Future readers must be able to weight AI-generated
content against human-written content without guesswork.

## Anti-Patterns

- Closing a bug as `wontfix` without a reproduction attempt
- Skipping the disclaimer on any AI-generated comment
- Grilling a bug before trying to reproduce
- Re-litigating a rejected enhancement that already has an
  `.out-of-scope/` entry
- `needs-info` without the "Established so far / Still needed" split
- Hand-off to `ready-for-agent` without an acceptance-criteria brief

## Integration with Other Skills

- **`doc-align`** — grilling sub-loop in Step 4
- **`plan-backlog`** — `ready-for-agent` issues feed the backlog scan
- **`doc-prd`** — when an enhancement justifies a full PRD, hand off
- **`debug`** — bugs marked `ready-for-agent` route into `debug` when
  picked up
